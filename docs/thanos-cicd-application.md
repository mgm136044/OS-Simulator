# Thanos 스케줄링 알고리즘의 CI/CD 빌드 파이프라인 적용 분석

> 본 문서는 프로세스 스케줄링 시뮬레이터 프로젝트의 Thanos 커스텀 알고리즘(Task 8)을 CI/CD 빌드 파이프라인 도메인에 적용하는 학술적 분석 및 확장 설계를 다룬다.

---

## 1. 서론: CI/CD 스케줄링의 현재 한계

### 1.1 현행 CI/CD 스케줄러의 동작 방식

주요 CI/CD 시스템의 빌드 스케줄링은 대부분 단순 FIFO 또는 정적 우선순위 기반이다.

| 시스템 | 기본 스케줄링 | 한계 |
|--------|-------------|------|
| GitHub Actions | 우선순위 레벨 내 FIFO | 빌드 진행률 미고려, 동시 실행 수 제한 시 대기열 적체 |
| Jenkins | FIFO (Priority Sorter 플러그인으로 확장) | 정적 우선순위만 지원, 런타임 진행률 무시 |
| GitLab CI | FIFO + Fair Usage Quota | 프로젝트 간 공정성만 고려, 빌드 단위 최적화 없음 |
| CircleCI | FIFO + Concurrency Limit | 큐 내 재정렬 불가 |

### 1.2 핵심 문제: 완료의 불연속적 가치

CI/CD 빌드는 **"All-or-Nothing"** 특성을 가진다.

```
빌드 가치
  ↑
1 ┤                              ●  (100% 완료: 테스트 결과, 아티팩트, 배포)
  │
  │
0 ┤────────────────────────────○    (0~99%: 가치 없음)
  └──────────────────────────────→ 진행률
```

- 90% 완료된 빌드가 타임아웃되면 → 투입된 CPU 시간 전량 낭비
- 테스트 결과는 전체 수행 후에만 의미를 가짐
- 빌드 아티팩트는 마지막 단계에서 생성됨

이는 게임 서버 매칭과 구조적으로 동일하다 — 매칭이 완료되어야만 유저에게 가치가 발생하며, 절반만 매칭된 상태는 무의미하다.

### 1.3 기존 알고리즘의 CI/CD 적용 한계

| 문제 | FCFS | RR | SPN | 영향 |
|------|:----:|:--:|:---:|------|
| **Convoy Effect** | ✗ | ○ | ○ | 30분 통합 테스트가 2분 린트 체크를 블로킹 |
| **기아(Starvation)** | △ | ○ | ✗ | 장기 빌드가 무한 대기 (SPN) |
| **진행률 인식** | ✗ | ✗ | ✗ | 90% 완료 빌드 = 0% 빌드 동일 취급 |
| **매몰 비용 최소화** | △ | ✗ | △ | 선점 시 이미 투입된 자원 낭비 |
| **예측 불필요** | ○ | ○ | ✗ | SPN은 빌드 시간 사전 예측 필요 |

> ✗: 해결 못 함, △: 부분 해결, ○: 해결

**핵심 공백:** 기존 어떤 알고리즘도 "빌드 진행률"을 스케줄링 결정에 반영하지 않는다.

## 2. 시스템 모델: 수학적 정의

### 2.1 기본 정의

CI/CD 빌드 스케줄링 시스템을 다음과 같이 형식화한다.

**빌드 집합:**

$$J = \{J_1, J_2, \ldots, J_N\}$$

각 빌드 $J_i$는 튜플 $(a_i, b_i, s_i)$로 정의된다:
- $a_i$: 도착 시간 (push/PR 이벤트 발생 시각)
- $b_i$: 총 빌드 시간 (burst time)
- $s_i$: 빌드 스테이지 수

**러너 집합:**

$$R = \{R_1, R_2, \ldots, R_M\}$$  (M개의 CI 러너, 시뮬레이터의 프로세서에 대응)

**시간 t에서의 상태:**
- $r_i(t)$: $J_i$의 잔여 빌드 시간
- $\pi_i(t) = 1 - \frac{r_i(t)}{b_i}$: 진행률 (0 ≤ π ≤ 1)
- $\sigma$: 타임 퀀텀 (러너 할당 단위 시간)

### 2.2 Thanos CI/CD 스케줄링 규칙

**부스트 조건:**

$$\text{BOOST}(J_i, t) \iff r_i(t) \leq \frac{b_i}{2} \land J_i \notin \mathcal{B}$$

여기서 $\mathcal{B}$는 이미 부스트된 빌드의 집합이다.

**부스트 동작:**
- $J_i$를 Ready Queue 최상단(front)으로 이동
- $\mathcal{B} \leftarrow \mathcal{B} \cup \{J_i\}$

**스케줄링 루프 (단일 러너):**

```
매 퀀텀 σ마다:
  1. ReadyQueue.popleft() → 현재 빌드 J_i 선택
  2. J_i를 min(σ, r_i) 만큼 실행, current_time 갱신
  3. IF r_i(t) = 0: J_i 완료 처리
  4. ELIF BOOST(J_i, t): ReadyQueue.appendleft(J_i)  ← 부스트
  5. ELSE: ReadyQueue.append(J_i)  ← 일반 RR
  6. arrival_time ≤ current_time인 빌드를 ReadyQueue에 추가
```

> **큐 처리 순서:** 현재 빌드의 재큐잉(3-5단계)이 신규 도착 처리(6단계)보다 **먼저** 수행된다. 이는 Task 8 명세의 trace/테스트 기대값과 일치하는 순서이며, 비선점 구간에서 이미 실행 중이던 빌드가 동시 도착 빌드보다 큐 앞쪽에 위치하도록 보장한다.

### 2.3 비용 함수

**낭비 비용 (Waste Cost):**

빌드 $J_i$가 시간 $t$에 타임아웃 또는 취소될 경우:

$$W(J_i, t) = b_i - r_i(t) = \text{이미 소비된 CPU 시간}$$

Thanos는 $\pi_i(t) \geq 0.5$인 빌드를 우선 완료하므로, 높은 $W$를 가진 빌드의 타임아웃 확률을 감소시킨다.

**개발자 대기 비용 (Developer Wait Cost):**

$$D(J_i) = c_i - a_i = \text{Turnaround Time}$$

개발자의 피드백 루프는 $D(J_i)$에 직접 비례한다. Thanos는 절반 이상 완료된 빌드를 빠르게 마무리하여 $D$의 분산을 줄인다.

## 3. 시뮬레이션 시나리오: CI/CD 빌드 큐

### 3.1 시나리오 설정

**"월요일 오전 러시"** — 제한된 러너에 다양한 빌드가 몰리는 상황.

| 빌드 | 도착(AT) | 소요(BT) | 유형 |
|------|:-------:|:-------:|------|
| J1 | 0 | 3 | Hotfix: 린트 + 단위 테스트 |
| J2 | 1 | 5 | Feature-A: 통합 테스트 |
| J3 | 3 | 2 | 문서 빌드 |
| J4 | 5 | 4 | Refactor: 단위 테스트 |

- 러너: 1대 (자원 제약 상황)
- 타임 퀀텀: σ = 2

> 이 시나리오는 본 프로젝트 Task 8의 게임 서버 검증 케이스(P1-P4)와 동일한 파라미터를 사용한다. 도메인이 달라도 Thanos의 수학적 행동이 동일함을 보이기 위함이다.

### 3.2 알고리즘별 실행 추적

#### FCFS

```
Runner: [J1: 0─3][   J2: 3────────8   ][J3: 8─10][ J4: 10────14 ]
         ███████  ████████████████████  ████████  ██████████████
```

| 빌드 | 완료 시각 | TT | WT |
|------|:--------:|:--:|:--:|
| J1 | 3 | 3 | 0 |
| J2 | 8 | 7 | 2 |
| J3 | 10 | 7 | 5 |
| J4 | 14 | 9 | 5 |
| **평균** | | **6.50** | **3.00** |

문제: J3(문서 빌드, BT=2)가 J2(BT=5) 뒤에서 5단위 대기 — **Convoy Effect**.

#### RR (σ=2)

```
t=0-2:  J1 실행(rem=1) → 재큐잉 후 J2(AT=1) 도착
t=2-3:  J1 실행(rem=0) ✓완료 → J3(AT=3) 도착
t=3-5:  J2 실행(rem=3) → 재큐잉 후 J4(AT=5) 도착
t=5-7:  J3 실행(rem=0) ✓완료
t=7-9:  J2 실행(rem=1) → 재큐잉
t=9-11: J4 실행(rem=2) → 재큐잉
t=11-12: J2 실행(rem=0) ✓완료
t=12-14: J4 실행(rem=0) ✓완료
```

| 빌드 | 완료 시각 | TT | WT |
|------|:--------:|:--:|:--:|
| J1 | 3 | 3 | 0 |
| J2 | 12 | 11 | 6 |
| J3 | 7 | 4 | 2 |
| J4 | 14 | 9 | 5 |
| **평균** | | **6.75** | **3.25** |

문제: J2는 누적 CPU 4단위를 소비하고도 t=12에야 완료. J4도 t=14까지 걸림.

#### SPN (비선점)

```
t=0-3:  J1(BT=3) 실행 ✓완료
t=3-5:  J3(BT=2) 선택 — J2(5)보다 짧음 ✓완료
t=5-9:  J4(BT=4) 선택 — J2(5)보다 짧음 ✓완료
t=9-14: J2(BT=5) 최후 실행 ✓완료
```

| 빌드 | 완료 시각 | TT | WT |
|------|:--------:|:--:|:--:|
| J1 | 3 | 3 | 0 |
| J2 | 14 | 13 | **8** |
| J3 | 5 | 2 | 0 |
| J4 | 9 | 4 | 0 |
| **평균** | | **5.50** | **2.00** |

문제: ATT는 최적이지만 **J2가 8단위 기아**. Feature-A 개발자는 피드백을 13단위 후에야 받음.

#### Thanos (σ=2)

```
t=0-2:  J1 실행(rem=1). half=1.5 → 1≤1.5 ✓ BOOST!
t=2-3:  J1 실행(부스트, rem=0) ✓완료
t=3-5:  J2 실행(rem=3). [t=3: J3 도착]
t=5-7:  J3 실행(rem=0) ✓완료  [t=5: J4 도착]
t=7-9:  J2 실행(rem=1). half=2.5 → 1≤2.5 ✓ BOOST!
t=9-10: J2 실행(부스트, rem=0) ✓완료
t=10-12: J4 실행(rem=2). half=2 → 2≤2 ✓ BOOST!
t=12-14: J4 실행(부스트, rem=0) ✓완료
```

| 빌드 | 완료 시각 | TT | WT |
|------|:--------:|:--:|:--:|
| J1 | 3 | 3 | 0 |
| J2 | 10 | 9 | 4 |
| J3 | 7 | 4 | 2 |
| J4 | 14 | 9 | 5 |
| **평균** | | **6.25** | **2.75** |

### 3.3 비교 분석

| 지표 | FCFS | RR | SPN | **Thanos** |
|------|:----:|:--:|:---:|:----------:|
| 평균 Turnaround | 6.50 | 6.75 | **5.50** | 6.25 |
| 평균 Waiting | 3.00 | 3.25 | **2.00** | 2.75 |
| 최대 Waiting | 5 | 6 | **8** | 5 |
| 기아 발생 | 없음 | 없음 | **있음(J2)** | 없음 |
| Makespan | 14 | 14 | 14 | 14 |

**분석:**

1. **Thanos vs RR:** ATT 7.4% 개선 (6.75→6.25), AWT 15.4% 개선 (3.25→2.75). 부스트가 절반 이상 완료된 빌드를 빠르게 마무리시켜 후속 빌드의 대기도 단축. 특히 J2의 WT가 6→4로 33% 감소.

2. **Thanos vs SPN:** ATT는 SPN이 우세하지만, SPN의 최대 WT=8 vs Thanos의 최대 WT=5. SPN은 J2 개발자에게 **8단위 기아**를 유발 — CI/CD에서 이는 "내 PR 빌드가 영원히 안 돌아간다"는 개발자 경험 문제로 직결.

3. **Thanos vs FCFS:** ATT 3.8% 개선에 더해, FCFS의 Convoy Effect를 해소 (J3의 WT: 5→2).

4. **Thanos의 고유 장점:** 유일하게 "기아 없음 + 진행률 기반 최적화"를 동시 달성.

## 4. 심화 분석: CI/CD 특유의 고려사항

### 4.1 빌드 진행률 추정 문제

게임 서버에서는 매칭 진행률이 명확하지만, CI/CD에서는 잔여 시간($r_i$)을 정확히 알기 어렵다. 이를 해결하는 세 가지 접근:

**방법 1: 스테이지 기반 추정**

CI/CD 빌드는 일반적으로 단계(stage)로 구성된다:

```
checkout → build → test → deploy
  (1/4)    (2/4)   (3/4)   (4/4)
```

진행률을 스테이지 완료 비율로 근사:

$$\hat{\pi}_i(t) = \frac{\text{완료된 스테이지 수}}{\text{전체 스테이지 수}}$$

부스트 조건: $\hat{\pi}_i(t) \geq 0.5$ (절반 이상의 스테이지 완료 시)

**방법 2: 이력 기반 추정**

동일 파이프라인의 과거 N회 실행 평균을 $\bar{b}_i$로 사용:

$$\hat{r}_i(t) = \max(0, \bar{b}_i - (t - \text{start}_i))$$

**방법 3: 하이브리드**

스테이지 가중치를 이력으로 학습:

$$\hat{\pi}_i(t) = \sum_{k=1}^{K} w_k \cdot \mathbb{1}[\text{stage}_k \text{ completed}]$$

여기서 $w_k$는 스테이지 $k$의 실행 시간 비중 (이력 평균).

> 시뮬레이터에서는 BT가 미리 주어지므로 정확한 잔여 시간을 사용하지만, 실제 시스템에서는 위 방법으로 근사한다.

### 4.2 타임아웃과 매몰 비용

클라우드 CI 러너는 실행 시간 제한이 있다 (GitHub Actions: 6시간, CircleCI: 5시간). 타임아웃 시 매몰 비용이 발생한다.

**시나리오: 글로벌 데드라인 T=11인 경우**

3절의 시뮬레이션에서, t=11 시점에 미완료 빌드는 타임아웃되며 투입된 CPU 시간이 전량 낭비된다.

각 알고리즘의 t=11 시점 상태 (3.2절 trace 기준):

| 알고리즘 | 완료 빌드 | 타임아웃 빌드 (t=11 시점 상태) | 낭비 CPU |
|---------|:--------:|:---------------------------:|:-------:|
| FCFS | J1, J2, J3 (3/4) | J4: t=10에서 시작, 1단위 실행 후 미완료 | **1** |
| RR | J1, J3 (2/4) | J2: 누적 4단위 실행(3-5, 7-9), rem=1 / J4: 2단위 실행(9-11), rem=2 | **6** |
| SPN | J1, J3, J4 (3/4) | J2: t=9에서 시작, 2단위 실행 후 미완료 | **2** |
| **Thanos** | **J1, J2, J3 (3/4)** | J4: t=10에서 시작, 1단위 실행 후 미완료 | **1** |

**핵심 관찰:**

RR에서 J2는 t=11 시점에 rem=1 — **단 1단위만 더 실행하면 완료**되지만 타임아웃된다. 이미 4단위의 CPU를 소비한 상태이므로, 이 4단위가 전량 낭비된다. Thanos는 J2를 t=9에서 부스트하여 t=10에 완료시켰으므로, 동일한 4단위 투자가 성공적으로 회수된다.

| 지표 | FCFS | RR | SPN | **Thanos** |
|------|:----:|:--:|:---:|:----------:|
| 완료율 | 75% | **50%** | 75% | **75%** |
| 낭비 CPU | 1 | **6** | 2 | **1** |
| J2 완료 여부 | ✓ | ✗ (rem=1) | ✗ (rem=3) | **✓** |

**관찰:** 이 시나리오에서 부스트는 "이미 절반 이상 실행된 빌드의 완료를 타임아웃 전에 확보"하는 효과를 보인다. RR 대비 낭비 CPU가 83% 감소(6→1)했으며, 이는 부스트에 의한 J2 조기 완료가 직접적 원인이다. 다만, 이 결과는 특정 시나리오에 한정되며, 일반적 부등식 $E[W_{\text{Thanos}}] \leq E[W_{\text{RR}}]$의 성립 여부는 빌드 분포와 타임아웃 값에 따라 달라질 수 있다.

### 4.3 연쇄 해제 효과 (Cascading Release)

CI/CD에서 빌드 완료는 단순 종료가 아니다:

```
빌드 완료 → 러너 해제 → 대기 빌드 시작 가능
         → 아티팩트 캐시 활용 가능 (후속 빌드 BT 감소)
         → 머지 큐 진행 가능
         → 배포 파이프라인 트리거 가능
```

Thanos의 부스트로 빌드 하나가 1단위 빨리 완료되면, 그 러너를 기다리던 다른 빌드도 1단위 빨리 시작한다. 이 효과가 큐 전체에 전파된다.

**수식 표현:**

빌드 $J_i$의 완료가 $\Delta t$만큼 앞당겨지면, $J_i$ 이후의 모든 빌드 $J_k$ ($k > i$)의 시작 시간도 최대 $\Delta t$만큼 앞당겨진다:

$$\forall k > i: \Delta D(J_k) \leq \Delta t$$

N개의 대기 빌드가 있을 때, 총 대기 시간 감소:

$$\Delta D_{\text{total}} \leq N \cdot \Delta t$$

### 4.4 멀티 러너 환경

실제 CI/CD는 다수의 러너를 운용한다. 시뮬레이터의 멀티코어(P core + E core)와 직접 대응:

| 시뮬레이터 | CI/CD |
|-----------|-------|
| P core (2x 성능, 3x 전력) | 고성능 러너 (large, GPU 등) |
| E core (1x 성능, 1x 전력) | 표준 러너 |
| 프로세스 | 빌드 Job |
| 소비전력 | 러너 비용 ($/분) |

Thanos를 멀티 러너에 적용할 때:
- **빈 러너 발생 시:** Ready Queue에서 부스트된 빌드를 우선 할당
- **러너 유형 선택:** 부스트된 빌드를 고성능 러너에 할당하면 완료 가속 (비용 대비 효율 최적화)

### 5.2 기대 결과 예측

10개 빌드, BT 합계 = 41, 러너 2대이므로 이론적 최소 makespan = ceil(41/2) = 21.

| 지표 | FCFS | RR | SPN | Thanos (예측) |
|------|:----:|:--:|:---:|:------------:|
| Makespan | ~21 | ~21 | ~21 | ~21 |
| 평균 TT | 높음 | 높음 | 낮음 | **중-낮음** |
| 최대 WT | 중간 | 높음 | **매우 높음** | 중간 |
| 기아 빌드 수 | 0 | 0 | **2-3개** | 0 |
| 타임아웃 위험(T=20) | 낮음 | **중간** | 낮음 | **낮음** |

**핵심 예측:** 빌드 수가 증가할수록 Thanos의 부스트 효과가 누적되어, RR 대비 ATT 개선율이 단일 러너(7.4%)보다 높아질 것으로 기대된다. 부스트된 빌드가 빠르게 완료 → 러너 해제 → 대기 빌드 시작의 연쇄 효과 때문이다. 다만 이는 가설이며, 시뮬레이터 구현 후 실제 수치로 검증이 필요하다.

## 6. Thanos가 CI/CD에 특히 적합한 이유

### 6.1 게임 서버 매칭 vs CI/CD 빌드 — 구조적 유사성

| 속성 | 게임 매칭 서버 | CI/CD 빌드 | 대응 |
|------|-------------|-----------|--------|
| 작업 단위 | 매칭 요청 | 빌드 Job | 프로세스 |
| 처리 자원 | 매칭 서버 스레드 | CI 러너 | 프로세서 |
| 다양한 BT | 솔로큐(짧음) vs 5인 파티(김) | 린트(짧음) vs E2E(김) | ○ |
| 공정성 요구 | 모든 유저 일정 시간 내 매칭 | 모든 PR 일정 시간 내 빌드 | ○ |
| 완료 우선 | 절반 매칭 → 이탈 방지 | 절반 빌드 → 낭비 방지 | ○ |
| 이탈/타임아웃 | 유저 매칭 취소 | 러너 시간 초과 | ○ |
| **진행률 관측** | **정확 (매칭 조건 충족 수)** | **추정 필요 (4.1절 참조)** | **△** |

두 도메인은 스케줄링 구조가 유사하여, 시뮬레이터에서 동일 파라미터로 비교 실험이 가능하다. 단, CI/CD에서는 정확한 잔여 시간($r_i$)을 직접 관측할 수 없고 추정에 의존하므로(4.1절), 부스트 판정 시점에 오차가 발생할 수 있다. **시뮬레이션 결과는 추정이 정확할 때의 상한(upper bound) 성능**이며, 실제 CI/CD 적용 시에는 추정 오차에 의한 성능 저하를 고려해야 한다.

### 6.2 세 가지 핵심 적합 조건

Thanos가 효과적인 시스템의 필수 조건:

| 조건 | CI/CD에서의 충족 | 비고 |
|------|----------------|------|
| **1. 완료의 불연속적 가치** | 빌드 99% = 가치 0. 100% = 테스트 결과 + 아티팩트 + 배포 | 완전 충족 |
| **2. 매몰 비용의 실체** | 중단 시 CPU 시간, 캐시 워밍, 네트워크 전송 전량 낭비 | 완전 충족 |
| **3. 공정성 제약** | 특정 브랜치/개발자만 우선 불가, 모든 빌드가 결국 완료 필요 | 완전 충족 |
| **4. 진행률 관측 가능성** | 정확한 $r_i(t)$ 관측 불가, 추정에 의존 (4.1절) | **부분 충족** |

조건 1-3은 CI/CD에서 명확히 성립하며, 이는 Thanos 적용의 **구조적 전제 조건**을 만족한다. 다만 조건 4(진행률 관측)가 추정에 의존하므로, 시뮬레이션에서 관찰된 성능 이점이 실제 환경에서 그대로 재현될지는 추정 정확도에 달려 있다. CI/CD는 Thanos의 **유망한 적용 후보 도메인**이지만, 진행률 추정기의 실측 검증 전까지는 가설 단계이다.

### 6.3 기존 스케줄링 대비 Thanos의 차별적 가치

```
              공정성 높음
                 ↑
            RR ──┼── Thanos
                 │      ↗
                 │    /
                 │  /  (유일한 조합)
                 │/
FCFS ────────────┼──────────→ 효율성 높음
                 │
                 │
            HRRN ┼── SPN
                 │
              공정성 낮음
```

Thanos는 공정성-효율성 2차원 공간에서 **기존 알고리즘이 도달하지 못한 영역**을 점유한다:
- RR 수준의 공정성 (기아 방지)
- SPN에 근접한 효율성 (진행률 기반 최적화)
- 예측 불필요 (SPN/SRTN과 달리 BT 사전 정보 불필요 — 실행 중 관측만으로 부스트 판단)

### 7.2 시뮬레이터에서의 검증 방법

본 프로젝트의 시뮬레이터로 CI/CD 시나리오를 직접 검증할 수 있다:

1. **3.1절 시나리오 입력:** J1(0,3), J2(1,5), J3(3,2), J4(5,4)
2. **알고리즘별 실행:** FCFS, RR(σ=2), SPN, Thanos(σ=2)
3. **결과 비교:** "전체 비교" 기능으로 Gantt 차트 + 메트릭 테이블 확인
4. **멀티코어 확장:** E core 2개 설정으로 5.1절 시나리오 검증

### 7.3 향후 연구 방향

- **진행률 추정 오차의 민감도 분석:** 4.1절의 세 가지 추정 방법에 대해, 추정 오차율(예: ±10%, ±20%)이 부스트 타이밍과 최종 성능 지표에 미치는 영향을 정량화. 이는 실제 CI/CD 적용 가능성을 판단하는 핵심 실험이다.
- **적응형 부스트 임계값:** 고정 50% 대신, 빌드 유형별 최적 임계값 탐색
- **가중 부스트:** 빌드 중요도(main branch vs feature branch)에 따른 부스트 강도 조절
- **실제 CI/CD 데이터 검증:** GitHub Actions 로그를 기반으로 한 시뮬레이션 재현 — 실제 빌드 시간 분포에서 Thanos의 이점이 유지되는지 확인
